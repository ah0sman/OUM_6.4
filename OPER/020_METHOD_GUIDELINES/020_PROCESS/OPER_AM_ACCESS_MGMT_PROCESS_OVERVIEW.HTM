[AM] ACCESS MANAGEMENT

From an ITIL perspective, “the purpose of access management is to provide the right for users to be able to use a service or a group of services.”1 “Access management is effectively the execution of the policies in ISM, in that it enables the organization to manage the confidentiality, availability and integrity of the organization’s data and intellectual property.”1

Back to Top

OBJECTIVES

From ITIL perspective, “the objectives of the access management process are to:

  • Manage Access to services based on policies and actions defined by ISM.
  • Efficiently respond to requests for granting access to services, changing access rights or restricting access, ensuring that the rights being provided or changed are properly granted.
  • Oversee access to services and ensure rights being provided are not improperly used.”

“Access controls are designed to limit access to the operating system of the Environment to approved named user accounts and to limit access to database passwords through the use of roles and policies. Oracle uses a combination of Tools and manual procedures to assist in controlling operating system and database access.”2

APPROACH

From OMCS perspective, Access management covers for:

  • My Oracle Support access
  • Hosted servers access
  • User administration

From OMCS perspective, the delivery team owns, coordinates and collaborates with the customer for enabling proper access to customer/implementer and periodic user audits.

This segment describes basic principles, recommendations and activities to be performed within Access Management process.

Basic concepts:

  • "Access The service functionality or data that a user is entitled to use
  • Identity Information that distinguishes a user; each identity is unique
  • Rights The settings that enable a user to access a service
  • Services or service groups Access is usually granted to groups of services, rather than to individual services
  • Directory services A tool that is used to manage access and rights."1

OMCS delivery team is responsible to provide to customers documentation and training with regards to access management process.

“Oracle Cloud Services employs an automated workflow system for Customer and/or its Implemented to submit User account requests. The system routes submitted requests to the appropriate manager(s) fpr approval. If a request is approved, an account provisioning tool creates the necessary User accounts on the relevant system(s).”2

The My Oracle Support Access Control page provides the functionality of managing Authorized Approvers and access to hosted servers. Through MOS the customer can:

  • Request access to customer CSI ( This action is enabling customer user to view and create Service requests and/or Request for Changes)
  • Request new OS accounts/rights ( This action is enabling customer user to log to hosted servers)
  • Reset password
  • Reset access policy
  • View history request
  • Add / remove authorized approvers (This action can only be performed by the Customer User Administrator)

The Customer User Administrator (CUA) acts as an interface between customer/implementer user access requests and OMCS delivery team. In short, the steps of the process are:

  • Any customer user who needs access should contact the CUA
  • CUA has the responsibility to review and centralize all requests and submit those to OMCS
  • OMCS delivery team sets up contacts in internal customer system
  • The user(s) can commence the self-service access process via MOS
  • OMCS delivery team follows up on internal systems approvals
  • Once all OMCS internal approvals are received the user receives email notification confirming access is granted.

It is recommended that OMCS team is conducting a periodical user audit and also that the customer is communicating new joiners, transfers, promotions and leavers in order to have the right level of access granted or revoked.
