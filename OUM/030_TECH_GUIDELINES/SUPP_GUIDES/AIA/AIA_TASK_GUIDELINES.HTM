ENTERPRISE ARCHITECTURE (EA) GUIDANCE

EA.075 - Develop Technology Reference Architectures

The SOA Reference Architecture describes a strategy on how to implement all types of services relevant for the business. It contains the chosen SOA Reference Architecture, a Service Provisioning Model, and a Roadmap to Service Implementation. The SOA Reference Architecture should cover services (EBS, ABCS) in all layers and granularities.

EA.110 - Develop Future State Enterprise Architecture

This task shows the "to-be" enterprise architecture using AIA Foundation Pack, including changes to existing architecture and newly identified reference architectures for future state realization.

BUSINESS REQUIREMENTS (RD) PROCESS GUIDANCE

RD.001 - Detail Business and System Objectives

This task scopes out the AIA foundation pack components considering the business objectives. It outlines the key business benefits of the system using the AIA Foundation Pack and also includes the priorities indicating which objectives are most important. The work product is used as reference throughout the project cycle.

RD.011 - Develop Future Process Model

The high-level Future Process Model documents the triggering events that drive the business areas that are to be automated and describes the response that the business makes to each of those events. This will help in designing the AIA components.

The more detailed Future Process Model defines the future business model in the form of integrated process flows built on the business processes supported by the new applications. The Future Process Model includes process flow diagrams of the events and business processes that the applications and the associated functions of the business area will support.

RD.030 - Develop Current Business Process Model

Examine the current business processes and practices to identify how the existing business system meets the current business requirements.

Also, you should perform the analysis of current business processes and functions during this phase.

RD.070 - Determine Audit and Control Requirements

In this task, you identify the high level requirements and policies that affect business and system security, control, and procedures.

  • Identify Participating Application Security Considerations / Requirements
  • Security Coverage / Considerations from the Customer Installed Base
Guidelines

Some possible questions from designers are:

  1. User Registration / termination policy
    1. Does your application have a need for integrating with specific Identity Provider infrastructure?
    2. Does your application have specific need pertaining to authentication of user identity?
    3. Does your application have any need for integrating with strong user authentication mechanisms such as smart cards, biometrics devices or dynamic passwords?
  2. Role-based access control policy – Key points
  3. Service audit and traceability policy – Key points
  4. Host and network administration policy – Key points
  5. Any specific needs on secure server-to-server connection
    1. Do you need to secure the web services communication exchanged between the service providers?
    2. Does your application have a need to integrate with 3rd party applications? Or in other words, are your applications sending and receiving messages over public network?
    3. Do you have specific security requirements regarding the server-to-server connection?
  6. Application administration policy – Key points
  7. Data encryption and signature verification policy
    1. Does your application have a specific need for data privacy?
    2. Does your application have sensitive data that you would not want that to be sent in clear text?
    3. Do you have any sensitive data that needs to be obfuscated in the logging?
    4. Does your application have specific SLA requirements for high availability, service continuity and recovery?

REQUIREMENTS ANALYSIS (RA) PROCESS GUIDANCE

RA.015 Develop Business Use Case Model

The business use case model encapsulates enterprise-level business process relationships. This has the design scope for enterprise. The focus is more on the operation of business rather than the computer systems, which helps in identifying the right candidates for PIPs, EBOs and EBSs.

This layout details the business processes that make up a process integration pack and every detail in the section, such as, requirements. Use cases must relate to this specific business process.

PIP Activity: Identify Integration Scenario(s)

AIA PIP Development Guidance:

Since this AIA activity applies to identifying integration scenario(s), it needs to include details on the functional flow, the initial requester, the service provider and the business events that would result in invocation of the integration.

RA.023 Develop Use Case Model

PIP Activity: Identify Operation(s) to be Performed on the Identified Object(s)

PIP Activity: Identify Enterprise Business Objects and Messages

AIA PIP Development Guidance:

PIP Activity: Identify Enterprise Business Service Operations

PIP Activity: Identify Participating Applications and their Roles Requesters/Service Providers

For AIA, the identified Enterprise Business Object(s) (EBO) should incorporate all of the requirements needed by the integration. The identified Enterprise Business Message(s) (EBM) serve as the message payload definition for the identified service operations. Each of the identified EBOs will have an Enterprise Business Service (EBS) to expose CRUD operations as well as any other supporting operations. Each of the actions present in the associated EBM will be implemented as a service operation in EBS. Creation of the EBS and the implementation of all of the service operations will be one of the critical tasks in the implementation of the end-to-end integration scenario. In situations where the EBS exists, check whether all of the actions necessary for acting on the EBO pertaining to this integration scenario have been implemented as service operations; if not, create a custom EBS to implement the additional service operations and upgrade proof the implemented extensions.

RA.024 Develop Use Case Details

PIP Activity: Identify the Requester Application Request Operations and Messages

PIP Activity: Identify the Provider Application Service Capabilities and Messages

AIA PIP Development Guidance:

For AIA, the identified Application Business Messages and Operations for the Requester and Provider Applications should incorporate all of the requirements needed by the integration. The identified Application Business Messages serve as the message payload definition between the application and the implemented AIA artifacts, while the identified Application Business Operations serve as the service contract definition between the application and the implemented AIA artifacts.

DESIGN (DS) PROCESS GUIDANCE

DS.090 Design Data

PIP Activity: Define Enterprise Business Objects and Messages

Wherever possible, design an EBO with reusable objects by using object types that are inherited by the different common objects in which they are referenced. An example would be an Address definition type. If your implementation requires customizing this address format by adding a third address line, the modification of the Address definition type automatically affects the addresses referenced in EBOs. This design philosophy significantly reduces the design, development, and maintenance of common objects. The Enterprise Business Message (EBM) represents the specific content of an EBO needed for performing a specific activity. For example, an invoice might be used in three contexts – process, cancel, and update. The context for processing the invoice might warrant the presence of almost all of the elements present in the EBO. However, canceling the invoice might only need to identify the invoice instance to be cancelled. Assembling a set of common components and EBO-specific business components creates the context-specific EBO definitions. These context-specific EBO definitions are then used in the appropriate context-specific enterprise business messages (EBMs). In this scenario, the process-specific invoice definition would be part of the ProcessInvoice EBM and the cancel-specific invoice definition would be a part of the CancelInvoice EBM. These EBMs can be used either as the request or response parameters.

DS.120 Design Services

AIA Existing PIP Guidance:

PIP Activity: Review/Extend operation definitions and interaction patterns in identified Enterprise Business Services to address all implementation requirements.

PIP Activity: Extend Enterprise Business Services

PIP Activity: Extend Enterprise Business Flows or Composite Business Process

PIP Activity: Extend Application Business Connector Services

AIA PIP Development Guidance:

Define Service Patterns for Business Service Operations - In the context of AIA, most operations across applications are required to be modeled using Asynchronous interaction patterns (Fire and Forget, Request/Delayed Response) with the lone exception being the Query operation. Synchronous interaction is not a generally recommended AIA interaction pattern and may warrant a justifiable business reason for its adoption.

Design Enterprise Business Services - In the context of AIA, this involves a combination of design (Defining and Creating the Contract) and implementation (Implementing and Invoking the Enterprise Business Services) activities. For the Entity EBS, use the WSDLs from the Enterprise Service Library of the Foundation Pack. For the Process EBS, use the TemplateEBS.wsdl available in the Foundation Pack and create a Process EBS WSDL.

Design Enterprise Business Flows - In the context of AIA, this involves a combination of design (Defining and Creating the Contract) and implementation (Implementing and Invoking the Enterprise Business Flow as a BPEL Service) activities. The EBF is used to implement a business activity or a task that involves leveraging capabilities available in multiple applications. An EBF strings together a set of capabilities available in applications to implement a coarse-grained business activity or task and composes a new service that leverages existing capabilities. An EBF involves only system-to-system or service-to-service interactions and does not include any activity that requires human intervention. In a canonical integration, the EBF is an implementation of an EBS operation and calls other EBSs. An EBF never calls an ABCS or application directly. In other integration styles, the caller invoking the EBF can be an application or any other service.

Design Composite Business Process - In the context of AIA, this is a similar task to design of EBF. The difference is that a Composite Business Process (CBP) is a set of coordinated tasks and activities involving both human and system interactions. In AIA, CBPs are implemented, managed, and monitored as a single SOA composite using SOA BPEL along with SOA Mediator, SOA Human Workflow, and SOA Business Rules components. The BPEL components are used to drive the flow.

Design Application Business Connector Services - In the context of AIA, this involves a combination of analysis (Analyze the participating application integration capabilities, Choose the implementation style and technologies, Choose message exchange pattern) and design/implementation (Design, Construct and Test Application Business Connector Services) activities. An ABCS is transport neutral, PIP agnostic and is not bound to a specific application version. ABCS design decisions should therefore not be based on the PIP for which it is being built. For example, EBM Population should not be based on a target system or a PIP.

Design Adapter Services (SOAP entry points to AIA artifacts) - In the context of AIA, this involves a combination of analysis (Analyze the participating application integration capabilities, Choose the implementation style and technologies, Choose message exchange pattern) and design/implementation (Design, Construct and Test Adapter Services) activities. To promote interoperability, implementation flexibility and transport neutrality, AIA mandates the ABCS to be designed and implemented with a SOAP interface. This requirement necessitates an abstraction layer between the endpoint application and the ABCS artifact.

IMPLEMENTATION (IM) PROCESS GUIDANCE

IM.050 Implement Components

PIP Activity: Review/Extend implementation of the Enterprise Business Service (provider service registration, routing rules, CAVS enablement, etc.)

PIP Activity: Review/Extend implementation of the Requester Application Business Connector Service (header population, transformation, routing, error handling, invocation, etc.)

PIP Activity: Review/Extend implementation of the Provider Application Business Connector Service (header population, transformation, routing, error handling, invocation, etc.)

AIA Existing PIP Guidance:

PIP Activity: Revisit Custom XPATH Functions where applicable

PIP Activity: Revisit AIA service artifacts to work with CAVS

PIP Activity: Revisit AIA service artifacts for error handling and logging

PIP Activity: Revisit service properties in relevant AIA service artifacts

AIA PIP Development Guidance:

Enable the Requester and Provider Applications - To enable the requester and provider applications in the context of AIA, identify the service APIs that need to be invoked, obtain operation WSDL(s) and payload schemas and construct adapter services if required. The interaction mechanism with the applications can be one of the following: SOAP Web Service, Events/Queues and Adapters (JCA, Message Queue and Database). Obtain relevant details from the applications in situations where the interactions happen through an adapter, as these details will be used in adapter configuration resulting in WSDL creation. Care must be taken to ensure that the environment specific details are configured in relevant resource files on the server and not directly in the generated WSDLs.

It is highly recommended that participating applications making an outbound interaction in an asynchronous mode use message queues to publish the requests. This approach is recommended for scalability and a better end-user experience. Participating applications making an outbound interaction in a synchronous mode alone should send the requests via SOAP / HTTP.

EBS Implementation Tasks:
  1. Create ESB Projects with Routing Services using the AIA EBS WSDLs for the identified message exchange pattern.
  2. Create Routing Rules to route the request.
  3. Implement Error handling for logging and notification based on fault policies.
ABCS Implementation Tasks:
  • Establishing connectivity with the participating application
  • Content Validation (if required)
  • Message Enrichment (if required)
  • Message Transformation
  • Population of EBM Headers
  • Invocation of Enterprise Business Services and/or Application Services
  • Manage referencing of instance identifiers and static values
  • Error Handling and Logging
  • Extensibility
  • CAVS Enabling
  • Implement Security Context (if required)
  • Internationalization and localization of message content (if required)
EBF or CBP Implementation Tasks:
  • Create a WSDL following the naming standards
  • Implement as a BPEL process
  • Enable error handling and logging
  • Enable extensibility points

IM.080 Integrate Services

If there is a need to implement a business activity or a task that involves leveraging capabilities available in multiple applications then Enterprise Business Flow should be chosen as a service-to-service interaction. If an orchestration of a series of human and automated steps is required then it should be implemented as a Composite Business Processes.

TESTING (TE) PROCESS GUIDANCE

TE.010 Develop Testing Strategy

The purpose of this task is to define how testing should be performed in the various stages at the project, which tools should be used (if any) and how, what metrics should be collected, how test data should be produced and what testing environments should be used in the tests. The strategy should also define the acceptance criteria of the solution.

TE.015 Develop Integration Test Plan

The Integration Test Scenarios consists of the set of Integration Test Scenarios for each use case to be tested as well as the Integration Test Specifications and the corresponding Data Profiles for any additional integration tests that are to be performed.

TE.020 Develop Unit Test Scripts

Develop the script to test individual components like EBSs, EBFs and ABCSs.

TE.030 Perform Unit Test

Perform unit tests for individual components such as, EBSs and ABCSs.

TE.040 Perform Integration (End-to-End) Test

Perform end-to-end integration tests to verify if all involved systems and AIA components work as required.

TE.065 Perform Installation Test

Install all application components in the System Test Environment to test the installation routines and refine them for the eventual Production Environment. This task is done in parallel with setting up the testing environments

TE.082 Develop Acceptance Test Plan

Assist in the preparation of test scenarios for each use case to be tested by users during the Transition phase. They should verify that all involved systems and AIA components work as required.

TE.120 Support Acceptance Test

Support users in performing acceptance tests that involve AIA components.
