REQUIREMENTS ANALYSIS (RA) PROCESS GUIDANCE

RA.015 Develop Business Use Case Model

In this task, the business use cases that represent the business requirements are defined. Scenarios that further define the use case and the actors that interact with the system are also defined. In custom BI development, most of the use cases will be associated with obtaining and analyzing data.  The development of a Business Use Case Model through the validation of the Functional Prototype (IM.010) relates to creating use cases that match business requirements to ensure the business intelligence data model can support all high priority business use case.  This relates to the requirements modeling process in Data Warehouse Methodology.

RA.024 Develop Use Case Details

In Business Intelligence, there are many use cases to obtain data in order to perform analysis or create a report. Current process use cases may require obtaining data from multiple sources. This would actually provide at least two high level use cases for the BI environment; one to obtain the data from the multiple sources and integrate them into a BI environment, and the second use case documenting the business function that uses the data. Hence, the many "obtaining data" use cases are used to create or extend the logical and physical data models. Business rules would be documented and incorporated into the Extract, Transform, and Load (ETL) processes documented in a tool such as Oracle Warehouse Builder (OWB). Some common use cases may wind up being stored procedures, derived fields, or materialized views in the final BI architecture. Of course, this is part of the analysis process that is common to all methodologies. Please note that the business processes will be used to ensure the data model can provide information to the business functions (obtained from the requirements) documented as use cases.

RA.030 Validate Conceptual Prototype

The validation usually occurs at a work session or the results of the work session may be validated via minutes from the work session with accompanying diagrams.  It is not unusual for the Conceptual Prototype to be slightly altered as more detail requirements and system constraints are documented.

RA.035 Develop High-Level Software Architecture Description

In this task, you ensure the conceptual architecture can support the business requirements. As requirements in the form of use cases are better understood and prioritized, the architecture evolves from conceptual to a physical environment.

ANALYSIS (AN) PROCESS GUIDANCE

AN.050 Analyze Data
AN.060 Analyze Behavior

In these tasks, you analyze use cases documented during requirements gathering and ultimate organize the results into the Analysis Specification.  Use cases are associated with all the business and system processes (for example, adding a user to the environment, creating a purchase order, validating a payment was made to a particular vendor, etc.)  These tasks are repeated for each use case, in the order of their priorities in the MoSCoW List (Must Have, Should Have, Could Have, Won't Have). Use cases can be classified into broad types such as interface classes, control classes, and data access classes. An example of a use cases in an interface class could be the inbound update use cases and the outbound extract use cases.

The Analysis Specification is the collation and organization of the Data Analysis (AN.050) and the Behavior Analysis (AN.060) contains the analyzed use case that show how the user requirements documented as use cases can be satisfied by the application.  This can be done through diagrams, sequence of events, and user walk throughs.  For example, data access requirements (use cases) would use the data model to show how data would be obtained for reporting or analysis.  The requirement (use case) would then become a test case with expected results confirmed during user acceptance testing.  This would complete the cycle from requirements through tested result using the use case concept.

AN.110 Review Analysis Model

Approach

These reviews can assist with learning events such as understanding the modeling approach when creating dimensional models.  A new approach to analyzing data may generate tasks within the end-user training plan.  Minutes from these review sessions are critical to tracking changes especially if the changes result in formal change requests.

IMPLEMENTATION (IM) PROCESS GUIDANCE

IM.005 Develop Conceptual Prototype

The prototype diagrams are usually created during a work session and often documented in Visio.  The diagram shows the interfaces to the data warehouse environment, how areas such as the Daily Business Intelligence might be used, and how the end user will access the environment.

The Conceptual Prototype will be expanded and is the first step in an iterative process to gain concurrence to the functional and technical architecture needed to support the requirements.

IM.040 Implement Database

The DDL to create the database is usually generated from a design tool such as Oracle® Warehouse Builder (OWB).  In some cases such as Oracle® Siebel Analytics, the baseline physical model is extended to meet specific customer business requirements.

Implementing the database only generates the database structures and does not populate the tables with data.  For BI, update processes must be created or configured in the case of prebuilt processes.  Security at the data level such as Virtual Private Database (VPD) is usually included in the design.

IM.050 Implement Components

In many cases, the components can be generated using tools, such as Oracle Warehouse Builder (OWB). OWB can also be used as a design tool.  The Data Mapping (CV.027) is a primary input to the ETL process.

IM.070 Assemble Components

An example of assembling components is creating a common updated script that calls multiple ETL modules to load a set of data.  An example of data access may be using a Discoverer report as content for a portlet within a common portal area.

TESTING (TE) PROCESS GUIDANCE

TE.005 Determine Testing Requirements

Review the Business and System Objectives (RD.001) and the Technical Architecture Workshop Results (TA.010) to understand the business and system objectives and the architecture of the solution.

The Testing Requirements define the business requirements for testing the data warehouse solution. It defines test requirements according to their significance or criticality to the business.

  • Review the Technical Architecture Workshop Results (TA.010) to understand the technical risks in the solution. Some of them will become candidates for testing.
  • Review Quality Management Plan (QM.010), and/or the Quality Control and Reporting Process (QM.020), as well as the Manage Issue and Problem Management Process for background information on Problem Management procedures.
  • Work with ambassador users to identify the Testing Requirements and document them clearly. In the process, it may be required to understand the quality policy of the customer and try to conform to that.
Additional Prerequisites

The Technical Architecture Workshop Results (TA.010) contains requirements about the future technical architecture that should be reflected in the Testing Strategy.

TE.010 Develop Testing Strategy

Acceptance Criteria

One component of the Testing Strategy is specifying the Acceptance Criteria. In a custom BI engagement, the Acceptance Criteria defines the metrics required to obtain acceptance of the solution. It should include criteria to support successful implementation of the solution in production including:

  • documentation
  • extracted data
  • data access
  • data acquisition (ETL)
  • system management
  • solution administration
Test Scope and Approach (Objectives)
Unit Test

The objective of the Unit Test is to test the functionality of all the individual components that comprise the data warehouse solution, for example, ETL maps, error handling, report validations, calculations, security, data warehouse administration components, and adherence to standards. The Analysts typically perform this first level of testing in the data warehouse development environment during the Construction phase.  As in all structured approaches to testing, the Testing Process emphasizes early testing (and verification in general) at each appropriate level. Adequate review of the design effectively prevents defects at the component level. Where the component is visible to an ambassador user the user should review and validate the component interactively. Where the output from the component is not visible, additional testing by the developer is needed before the component can be integrated.

System Test

The objective of the System Test is to verify the functionality, security and other system requirements of the entire data warehouse system. System testing may include regression testing to verify whether existing components of data warehouse solution delivered through earlier iterations work properly with the components of the new iteration.

Systems Integration Test

The objective of the Systems Integration Test is to verify the integration of the data warehouse system with the external applications. This will be carried out in the test environment. During this test, following interface components will be tested.

Final Data Validation and Data Access Test

This test is carried out in the Production Environment. This is a final walkthrough done by the development team before it is delivered to the users. All the business critical reports are tested in this step before going live.

Test Environments

There are several things to consider when defining the required test environments, such as:

  • Review customer requirements for testing and release management for any specific environment requirements.
  • Consider sources systems, for example, testing business scenarios in source to validate correct capture and processing in data warehouse may require the source system to be controlled.

Business Critical Data Validation Test

When performing the Business Critical Data Validation you should identify the business critical data with the help of Information Analyst. This can be categorized by business or subject area. Also, identify the data access components (reports) that use these critical data elements. For each of these reports, do the following.

  • Review the test scenario deliverables (TE.035 – Integration Test Scenarios, TE.025 - System Test Scenarios and TE.090 - Systems Integration Test Scenarios) to identify the scenarios intended to test this critical measure. Run the report with the parameters taken from the script and note down the value shown by the report.
  • Validate the correctness of the data/calculation by verifying the report result against the expected value. The expected value is documented in the test scenarios.
  • In cases where identified test scenarios cannot be used for the validation test, create test cases of your own and document the steps in the column ‘Parameters for Validation’ in the template.
  • In cases where test scenarios can be used, make a reference to the test case ID in the column ‘‘Parameters for Validation’.
  • Record the results and update the status as pass/fail

When you have done this, you should consider the following:

  • Have all data elements been verified?
  • Is the list of elements complete?
  • Has customer overlooked any key elements which should be included?
Data Access Capabilities Test

When performing the Data Access Capabilities Test you should first understand the user profiles and role access definitions. Then list the critical data access components (reports) and list the user roles that have been provided access to this report. For each of these reports:

  • Test whether the report is accessible for this role. This can be performed by creating a dummy user and then assigning the user this particular role and then logging in as the dummy user and verifying if the report is accessible.
  • If the Fine grained Security solution is used in the project (for example, VPD) fill up the column ‘Data Valued allowed by the role’ with the textual description or range of data. Delete this column, if this is not applicable to your project.
  • Record the results.

When you have done this, you should consider the following:

  • Are elements accessible to reporting users?
  • Are tables and views accessible to the reports?

Data Acquisition - Data Load Strategy - Referential Data Refresh Approach

The best practice it to maintain the referential data (for example, list of values) from source systems as any other data. However, this is not always possible since the referential data might be hardcoded in source systems, the interface for referential data might not exists or there is simply no source system for a particular referential data. Therefore, document how the referential data is handled in the solution, how they are updated and who is responsible for the update. Consider creating a special area to store the referential data and use the area as if it was any other source system.

Data Acquisition - Data Load Strategy - Dimensional Table Refresh Approach

Provide the guidelines on refreshing dimensions. Include methods for detecting changes in dimensions, methods for implementing slowly changing dimensions and methods to update audit attributes.

A typical method for detecting changes is to compare the source data with the data in the data warehouse using set operations (minus, union) in order to prepare data set containing only the changes. This method can be used with both full and incremental extracts.

Data Acquisition - Data Load Strategy Component - Fact Table Refresh Approach Section

Provide the guidelines on refreshing fact tables. Include methods for maintaining table partitions and indexes, how to publish the newly loaded data to users and methods to update audit attributes.

Consider using the Partition Exchange Loading (PEL) method when updating fact tables. PEL is the preferred method for updating partitioned fact tables. Decide whether to use out of the box PEL support in OWB (with all its restrictions) or a custom approach.

If the system allows backward changes to fact tables describe methods how to handle and propagate the changes (for example, if it is necessary to load late arriving transactions, how to insert the transactions to a fact table, how to differentiate late arriving records from the other records, how to propagate late arriving records to summaries).

Data Quality - Objectives Component - Data Quality Overview Section

The following boilerplate text might be used to describe the data quality issues in the data warehouse. This text can be added to the Data Quality Overview section of the work product.

The best way how to answer “why is the data quality important” is to reverse the question - what happens when the data do not have required quality. There are many cases of incurred costs, lost credibility or missed business opportunities because of incorrect data. For example

  • Sending direct mails to incorrect addresses
  • Contacting dead customers
  • Contacting the same customer twice with the same offer
  • Offering products to customers that should not qualify
  • Refusing products to customers that should qualify
  • Restatement of financial statements
  • Wrong data sent to regulators

A data warehouse is especially vulnerable to poor data quality. Unlike legacy systems, the primary objective of the data warehouse is to provide data to knowledge workers and to analytical applications. If the data quality does not allow knowledge workers to use it for analysis or decision making they will stop using the system and it will fail.

A data warehouse is also usually the first system that consolidates data from many sources into a common data model. As a result it is much easier to discover the data quality issues in the data warehouse than it is in legacy systems.

So even if the data warehouse does not cause most of the data errors, it is in the vital interest of the data warehouse project to influence the source systems and information producers to improve the data quality.

There are many reasons why data can be incorrect. The following are some typical reasons for a data warehouse environment, ordered by the information flow sequence.

  • Information is incorrectly provided to data entry clerks.
  • Information is incorrectly entered into a source system.
  • Optional data are not entered.
  • Source system requires information that is not available.
  • Information is duplicated in the source system.
  • The same information in different source systems is inconsistent.
  • The source system incorrectly calculates derived data.
  • The data are incorrectly extracted from the source system.
  • The data are extracted from source systems at bad time.
  • The files are incorrectly transferred to the data warehouse.
  • The data are incorrectly processed in the data warehouse.
  • Users select incorrect data from the data warehouse.
  • Users do not understand data in the data warehouse.
  • Behavior of source systems changes over time but the rules and processes are not updated to reflect this change.

The previous paragraph clearly shows that the data quality is not a function of one system or one role. Instead, all the applications, architects, information producers and knowledge workers have to take responsibility for continually improving the data quality.

Therefore it is necessary for an organization to establish an environment that leads to sustainable improving of information quality. The data quality initiatives might bring some short term benefits but unless it is a continuing process the data quality will deteriorate over time.

Data Quality - Processes Component - Data Transfer Section

The following methods might be considered for ensuring the data transfer integrity for the specific data warehouse solution that should be included in the work product template for this task.

One method how to control the data transfer process is via “control files”.

Control files complement the data files. They contain information about data files within a file set, version of the file structure, effective date, number of records in the data files and (if no character conversion takes place) they could also contain number of characters and file checksum.

When data files are received and before any processing is done, the files are checked against the control files. If the data file does not correspond with the control file than the administrator is notified, the whole data file is considered corrupt and either the transfer or the data extraction must be repeated.

Another method is to structure the data files in a way that enables consistency checks of the files. The following rules can be used:

  • Each row is prefixed with a row type
  • One row type correspond to the file header, containing
    • File name
    • Effective date
    • File structure version
  • One row type correspond to the file footer, containing
    • Number of data rows within the file
    • Optionally other summary measures (e.g. sum of balances)

When data files are received and before any processing is done, the file structure is checked. The file must have the header row and the footer row, the effective date and file structure version should correspond to the expected values, the number of data rows and summary measures have to be correct. If the file does not meet the checks the whole file or file set is rejected.

All these methods can be used together to provide a maximum level of control over the data transfer process.

Data Quality - Processes Component - Audit and Control Section

The following text and example may be useful as a starting point for defining the audit and control methods for the specific data warehouse solution that should be included in the deliverable template for this task.

The following text describes where to put automatic the data quality checks:

Checkpoints in the ETL Process are automatic checks executed during the extraction, transformation and load process (ETL) that validate number of processed records. If the number of records does not correspond the administrator is notified and optionally the whole process is stopped until the problem is resolved. The typical checkpoints are

  • After Extraction – check number of extracted records.
  • After Transfer – check number of records before and after the transfer.
  • After Load – check number of records loaded.
  • After Allocation – check sum of allocated figures against the total.
  • After Aggregation – check the sum of detail data against the aggregation.

Number of records can be checked against the source (in case of 1:1 mappings), or, alternatively, it is possible for each data file to define a typical volume of data and the check is done against this volume. For example, we can define that the incremental extract should have between 0 and 50000 records or that the full extract should have more than 4 million records. If these conditions are not met than the administrator is notified.

Usually if the data does not meet these quality checks than the whole data set is considered corrupt and the ETL process is stopped until a detailed analysis is done and the process is either resumed or canceled.

The following methods can be used to report the data quality and to inform about the data (non) quality in the data warehouse:

Method Description
Exception Reports
Exception reports list the detail records that contain missing values or do not conform to the business rules. The exception reports are usually passed to process or data owners for correction.

Exception reports are usually produced ex-post, i.e. after the data is loaded into the data warehouse. It is therefore necessary for the system to re-process the corrected values after they are repaired in the source system.

If a number of records to be corrected is not large, than the exception records should be produced and distributed automatically. However, if the number of records to be corrected is significant, it is better to run a focused activity targeted at the quality improvement and only after the number drops to acceptable level to produce automatic exception reports.

For example, exception records can list active accounts with obsolete product, accounts with invalid interest rates or customers who should be merged with another customer (potential duplicates).
Summary Reports
Summary reports show percentage of records that meet quality criteria. Summary reports are dimensioned by source systems, products or organizational unit, in order to better understand the distribution of errors.

Summary reports provide the basis for regular monitoring of data quality. Similar to exception reports, summary reports can be defined only if the quality measures can be calculated using the data in the data warehouse.

Summary reports are produced automatically. It is useful to store results of summary reports in the database to support evaluation of data quality over time.

For example, summary reports can show percentage of correct addresses per address type and source system, percentage of duplicate customers per customer type or percentage of non classified current accounts in debit.

Summary reports can also be used to measure a completeness of data – a typical report will show percentage of records that do not have valid values per table column and source system. This report is also quite useful for knowledge workers.
Reconciliation Reports
Reconciliation reports compare data from the data warehouse with the same data from another surrogate data source. The reconciliation reports are used to measure difference between these two systems.

A typical example is to measure summary balances on customer accounts against the balances on GL accounts. Also it is possible to compare reports from source systems against the reports produced from the data warehouse.

CV.027 Perform Data Mapping

This task evolves during the development of a project.  It begins with the identification of the source systems and evolves into the actual data source of the attributes needed in the target system.  In many cases some type of transformation is needed to conform to the target system's architecture.  A byproduct of this task is the gap analysis matrix.  Some data elements needed in the target system may not be available.  These element may be entered manually through the application or in some cases a new function is required to maintain the "missing" source data elements.  The final source to target mappings with the transformations may be documented in a tool such as Oracle Warehouse Builder (OWB).

CV.040 Design Conversion Components (Initial Load)

Approach

For BI Data Acquisition, the primary objective of this task is to describe the specific ETL processes that enable the interfaces between source and target systems. It identifies all components of each ETL process including data complexities or constraints for conversions and interfaces. In addition test scenarios or cases are developed to support unit testing of each data acquisition component.

When the project team performs this task, most of the targets would have been designed and their mappings to source system objects identified (See CV.027, Data Mapping). High-level requirements and constraints of the interfaces would also be available from the the Data Acquisition, Conversion and Data Quality Strategy (CV.020). All components of the solution would be identified in the Final Platform and Network Architecture (TA.150).

This task involves understanding these requirements and translating them to specifications for individual mappings that extract, transform and load data between systems, and how they would fit in the overall architecture.

As far as possible, the deliverable template has been designed to make it independent of the ETL implementation tool, be it using custom PL/SQL scripts or using Oracle Warehouse Builder.

TR.020 Prepare Project Team Learning Plan

Approach

Many times, training of customer staff can be incorporated into normal activities.  For example, customers new to business intelligence may be familiar with normalized data models but may not be familiar with dimensional modeling.  Design reviews can incorporate learning activities.  Also, customer developers may assist by developing modules using tools such as Oracle Warehouse Builder.  Ensure the customer developers get training on new tools and account for their newly acquired skills in the project plan.  Ideally, the customer developers should reduce the time needed for development but be sure to factor in the need for Oracle consultants to help customer developers.
