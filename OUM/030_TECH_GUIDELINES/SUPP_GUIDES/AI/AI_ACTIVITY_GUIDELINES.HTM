INCEPTION

A.ACT.GBRI Gather Business Requirements - Inception

In a commercial off-the-shelf (COTS) application implementation, the “Gather Business Requirements - Inception " activity serves to detail the business and system objectives for the project and begins the process of gathering key business requirements and data needed to configure the application environment.

Approach

In a commercial off-the-shelf (COTS) application implementation, you may have access to pre-existing, multi-level business process models, which depict how the future application system supports the business functions being addressed by the project. If such process models are available, leverage them to help define elements of the Future Process Model (RD.011) and Business Use Case Model (RA.015) prepared during this activity.

A.ACT.CCPI Create Conceptual Prototype - Inception

In a commercial off-the-shelf (COTS) application implementation employing a requirements-driven approach, this activity is generally performed only when there is a need to gain further clarification of the business requirements represented in the use cases. This typically applies to those requirements that can only be satisfied through extensive and/or complex custom-built components identified during the pre-sales cycle and/or during the Inception Phase, which extend the functionality of the COTS system and/or integrate the COTS system with other COTS or legacy systems.

This activity is not normally performed in a commercial-off-the-shelf (COTS) application implementation employing a solution-driven approach that seeks to minimize custom extensions by promoting leading practice use of standard functionality. This activity is therefore not included in the pre-defined Solution-Driven Application Implementation View Work Breakdown Structure. However, if your solution-driven application implementation includes architecturally-significant custom extensions, you should consider including this activity in your project.

ELABORATION

B.ACT.DUCM Develop Use Model
B.ACT.DUCD Develop Use Case Details

In a commercial off-the-shelf (COTS) application implementation employing a requirements-driven approach, Use Case Models complement business process models in helping the implementation team gain a thorough understanding of business requirements. At a minimum, Use Case Models and Use Case Specifications should be developed for those requirements that can only be satisfied by custom-built components, which extend the functionality of the COTS system and/or integrate the COTS system with other COTS or legacy systems. Development of Use Case Models for all other business requirements is considered best practice, as they contribute to a better understanding of requirements and can be leveraged to prepare a number of other required work products downstream, such as test scenarios and business procedures documentation.

In a commercial off-the-shelf (COTS) application implementation employing a solution-driven approach, Use Case Models should be developed only for those requirements that can only be satisfied by custom-built components that extend the functionality of the COTS system and/or integrate the COTS system with other COTS or legacy systems.

B.ACT.DCCTSE Develop Custom Component Test Scripts - Elaboration

In a commercial off-the-shelf (COTS) application implementation, this activity is performed only for custom-built components that extend the functionality of the COTS system and/or integrate the COTS system with other COTS or legacy systems.

In an application implementation project, unit testing is generally performed in the Construction phase. However, it may be performed in the Elaboration phase for complex, higher risk custom extensions that have been identified during the pre-sales cycle and/or during the Inception phase, and where the prototypes of those complex custom extensions that have been developed during the Elaboration phase, will continue to be enhanced during the Construction phase.

B.ACT.DVCSP Develop and Validate Custom Software Prototypes

In a commercial off-the-shelf (COTS) application implementation, this activity allows for developing and validating the various prototypes of custom extensions to the applications, implementing the database extensions (where necessary) and performing a component review. Development and validation of the Functional Prototype(s) for custom extensions is generally limited to custom extensions that warrant development of a prototype at this stage of the project due to the technical complexity of the extension, or other risks associated with them.

It also serves to address the creation of a User Interface Standards Prototype environment for validation of any revisions made to the out-of-the-box application system user interface in order to meet the implementing organization’s requirements. This is generally applicable only to those COTS application systems, which enable tailoring/revision of the standard user interface via supported means, such as Oracle’s Siebel Tools.

In a commercial-off-the-shelf (COTS) application implementation employing a solution-driven approach that leverages pre-defined setup parameters and/or tools for rapidly establishing a working application environment, use this activity to develop and validate Functional Prototype(s) if there is a need to build functional prototypes of architecturally-significant custom extensions at this stage of the project.

B.ACT.PCCTE Perform Custom Component Testing - Elaboration

In a commercial off-the-shelf (COTS) application implementation, this activity is performed only for custom-built components that extend the functionality of the COTS system and/or integrate the COTS system with other COTS or legacy systems.

In an application implementation project, unit testing is generally performed in the Construction phase. B.ACT.PACDE Prepare to Acquire and Convert Data - Elaboration

In a commercial off-the-shelf (COTS) application implementation, this activity consists of producing the standards and completing the high-level (business object level) data mapping needed to confirm that the application software can support the customer organization's business processes and legacy data requirements. Conversion component design and development tasks are executed in a similarly named activity in the Construction phase.

CONSTRUCTION

C.ACT.FR Finalize Requirements

In a commercial off-the-shelf (COTS) application implementation, this activity is generally performed only when multiple iterations are planned for the Construction phase. In such cases, this activity is included in the second, and subsequent, iterations.

This activity is not normally performed in a commercial-off-the-shelf (COTS) application implementation employing a solution-driven approach. This task is therefore not included in the pre-defined Solution-Driven Application Implementation View Work Breakdown Structure. However, if your solution-driven application implementation includes two or more iterations within the Construction phase, you should include this task in your project.

PRODUCTION

E.ACT.RPP Resolve Production Problems

In a commercial off-the-shelf (COTS) application implementation, the "Resolve Production Problems" activity also involves the execution of the system management procedures, which are performed in perpetuity.

Approach

The ongoing system maintenance begins during this activity. Maintenance is typically not triggered by technical issues that the support or help desk address; this type of maintenance focuses on necessary and preventative actions.

Ongoing system maintenance should address the following:

  • timing of maintenance procedures
  • reporting unusual occurrences
  • interpreting system indicators
  • logging completion of maintenance work
  • adjusting the software and hardware for anticipated business changes

Verify that all normal, ongoing activities keep the system protected, purged, and operating as efficiently as possible. Make sure a log is available for recording all change events. Standard checklists can be incorporated for normal, standing tasks that must be performed (like backups and defragmentation of tablespace). In addition, use the log as a record when new users are added to the system.
