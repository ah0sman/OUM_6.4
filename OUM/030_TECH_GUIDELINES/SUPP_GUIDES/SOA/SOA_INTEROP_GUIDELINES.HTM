SERVICE-ORIENTED ARCHITECTURE (SOA) PROJECT DELIVERY SERVICE INTEROPERABILITY GUIDELINES

In order to provide or consume a "Business Service", the involved systems/partners must agree on the interaction pattern with the corresponding business semantics. Agreement must also exist for the message binding contracts and policy profiles for each service. This document provides guidelines, patterns and templates to capture all the necessary information related to service interoperability.

SERVICE SCOPE

This section introduces common high-level scenarios that illustrate Web Service solutions based on the scope of the service from the enterprise boundary perspective. This scope based Web Service scenario classification would be helpful to derive requirements as well as the possible patterns that can be utilized as a solution corresponding to a particular scenario.

Public Service

In this scenario, Web Services are exposed to the entire world and depending on the application, the security requirements will vary greatly ranging from no explicit security to high protection and prevention methods.

Intra-Enterprise Service

In this scenario, Web Services are exposed within an enterprise for other applications to consume.

Inter-Enterprise Service

In this scenario, Web Services are exposed to partners to conduct business-to-business transactions.

Federated Service

In this scenario, a set of Web Services is exposed by partners to collaborate and conduct a business transaction in a federated manner.

INTEROPERABILITY

Services are said to be interoperable only when the involved systems and partners understand both Business and Technology semantics.

Business Layer Policies

At the business layer, the domain object and the action are the 2 most important policies. Domain objects are nouns such as ‘Customer’, ‘Purchase Order’ etc. Actions are verbs such as ‘Create’, ‘Approve’ etc.

Business Message

The business messages are ‘domain objects’ (nouns) and are modeled using the following methods of achieving interoperability:

  • Distributed Point-to-Point - In this approach, each service (corresponding to a system) has its own proprietary semantic meaning and semantic translation is performed in a point-to-point manner local to a service. In other words, every service will be interoperating with every other service using its own internal transformation in a point-to-point manner.
  • Centralized Point-to-Point - In this approach, each service (corresponding to a system) has its own proprietary semantic meaning and semantic translation is performed in a point-to-point manner but centrally. In other words, every service will be interoperating with every other service using the central located transformation logic in a point-to-point manner.
  • Distributed Hub and Spoke (Canonical Data Model) - In this approach, each service (corresponding to a system) has its own proprietary semantic meaning but is mapped to a logical data model, which can be initiated as a physical federated model or a canonical data model. In other words, the Hub is virtual and is modeled as a canonical message and spokes will do the semantic transformation internally.
  • Centralized Hub and Spoke (Master Data Management) - In this approach, each service (corresponding to a system) has its own proprietary semantic meaning but is mapped to a single data model managed by a specialized application such as Master Data Management (MDM systems. In other words, the Hub is physical and is responsible for maintaining single version of truth.
Business Message Versioning

The versioning of business messages can be done using the following methods:

  • Type based: The versioning scheme is part of the domain object name. If the message is modeled using XSD, the target namespace for the domain object type could signify the version.
  • Instance based: The versioning information is sent as part of the message. If the message is modeled using XSD, then the corresponding XML message will carry the version information as part of an element or attribute.

A hybrid versioning by utilizing both type based (for major version number) and instance based (for minor version number) may be used as well.

Versioning could use the date format such as YYYY/MM/DD.

Business Transaction

The business transactions are ‘actions’ (verbs) that are modeled using the timing of interactions between the involved partners/systems/services.

Message Exchange Patterns

The best practice is to have one message exchange pattern, such as the following, per service and business processes should be used for orchestrating more than one service to model complex message exchange patterns.

Following are some of the common message exchange patterns for a coarse grained service.

  • Synchronous-Query-Response-Fault - This pattern should be used for scenarios where a response be returned for a query and the state of the system must not be changed.
  • Synchronous-Request-Response-Fault - This pattern should be used for scenarios where a response be returned for a request and the state of the system may be changed.
  • Synchronous-Request-Ack-Fault - This pattern should be used for scenarios where an acknowledgement be returned for a request to indicate that the provider received the request.
  • Asynchronous-One-Way - This pattern should be used for notification or quote like scenarios where the delivery reliability is not mandatory.
  • Asynchronous-Request-Response - This pattern should be used if the processing time of returning a response for the corresponding request is unpredictable.
  • Asynchronous-Request-Ack - This pattern should be used if the processing time of returning an acknowledgement for the corresponding request is unpredictable.
  • Asynchronous-Request-Ack-With-Timeout - This pattern should be used if the processing time of returning an acknowledgement for the corresponding request is unpredictable and if the consumer insists on getting the acknowledgement within a specified period of time.
  • Asynchronous-Request-Ack-With-Timeout-Notification - This pattern should be used if the processing time of returning an acknowledgement for the corresponding request is unpredictable and if the consumer would like to be notified when the acknowledgement is not received within a specified period of time.
  • Asynchronous-Request-With-Optional-Ack-With-Timeout - This pattern should be used if the processing time of returning an acknowledgement for the corresponding request is unpredictable and if the consumer insists on getting the acknowledgement optionally within a specified period of time.
  • Asynchronous-Request-Response-With-Optional-Ack-With-Timeout - This pattern should be used if the processing time of returning a response for the corresponding request is unpredictable and if the consumer insists on getting the response optionally within a specified period of time.
ACID Transaction

Some business services required to perform atomic transactions with ACID properties and are usually synchronous in nature.

The service will lock the appropriate resources while conducting the ACID transactions.

Above mentioned synchronous message exchange patterns should be used by the service if it requires conducting ACID transactions.

Long Running Transaction

Some services might require running for hours, days, months or years before concluding a business transaction.

The service will release the locks and hence isolation property of ACID is relaxed.

Above mentioned synchronous message exchange patterns should be used by the service if it requires conducting long running transactions.

Technology Layer Policies

At the technology layer, standards based policies related to transport, security etc. are important for higher interoperability.

Following are some of the common technology related components that are related to service interoperability.

Transport

The most common transport mechanisms used by service endpoints are HTTP or messaging middleware such as JMS, MQSeries etc.

In order to be interoperable, messaging/transport protocol version and headers need to be defined and agreed upon, such as ‘SOAPAction’ HTTP 1.0 header in case of SOAP based service invocation.

HTTP transport is not reliable, whereas messaging middleware such as JMS provides reliable delivery of messages. But HTTP is more prevalent and firewall friendly.

The best practice is to use HTTP for interacting with external partners and messaging middleware for internal partners/systems.

Batching

Some business processes such as monthly billing require batching capability instead of real-time interaction. In some other cases, business systems are inherently batch-oriented such as mainframe applications. In some cases, batching is used by small IT organizations to avoid higher IT and communication costs.

Services may have to support batch-oriented interaction in addition to real-time interaction. In order to support batching, the messaging model must support the following elements either implicitly or explicitly:

  • Batch-id
  • Sequence-number and
  • Batch-termination-flag
Reliability

The Technology Layer requires that the messaging systems of business partners that are involved in consuming and providing business services must support reliability in terms of Guaranteed Delivery, Guaranteed Processing and Visibility.

Guaranteed Message Delivery

Guaranteed Message Delivery is a mechanism to ensure that the message reaches the incoming message port of a service based on the QOS parameters such as only once or at-least once.

If HTTP is used as a transport, then higher-level protocols such as WS-ReliableMessaging must be used, if guaranteed message delivery is a requirement.

Guaranteed Message Processing

Guaranteed Message Processing is a mechanism to ensure that the message not only reaches the incoming message port but that it is correctly processed by the destination business service. It provides the inherent ability to ensure that a business service dialogue is completed to the confirmed satisfaction of all parties involved in that business transaction.

Guaranteed Business Processing

Guaranteed Business Processing can be achieved by sending a ‘Business Response or Business Acknowledgement’ message/MEP for every ‘Business Request’ message/MEP. Integration infrastructure does not need to implement business-level guaranteed processing functionality within itself, but will be implemented specific to business scenarios at the enterprise application level.

Visibility

Visibility gives a global view about the consistency and the current state of the various systems that were involved in business service execution.

Business users need visibility at the business document level and technical production support users need visibility at both business document level as well as at the messaging level including detailed logs and audits for debugging and managing. Any other visibility functionality other than related to guaranteed-delivery and guaranteed-processing need not be part of service infrastructure or integration application.

For example, capturing the number of successfully processed orders from a specific partner or system is a valid visibility requirement from a service point of view. Also, resending the messages that are not delivered due to, say, unavailability of the target system is part of the visibility/monitoring and management tool.

The following are some of the meta-data elements that would be captured and managed by the visibility tool:

  • Message ID – a globally unique ID
  • Related Message ID – correlation ID
  • From Partner Profile ID – points to the action specific partner profile
  • To Partner Profile ID - points to the action specific partner profile
  • Application Environment – environment context such as Production, Test etc.
  • Timestamp – created and received date time
  • Message Exchange Pattern – MEP used by the service
  • Business Document Context (Business Object IDs) – a composite key identifying the business document uniquely
  • Message Blob – sent or received message

The above attributes address only the technology level visibility and would be useful only to the system administrators. If business process level visibility in terms of KPI is required, then BAM component must be used.

Security

Hackers and cyber terrorists are launching security attacks with increasing frequency and sophistication. The traditional approach to security - a firewall combined with an anti-virus - is incapable of protecting from today’s advanced threats.

However, a formidable defense can be erected by implementing security using a layered approach as mentioned below. By selectively installing security measures on five levels (as listed below) within the IT environment, digital assets can be adequately protected and the risk of exposure to a catastrophic network breach can be greatly reduced.

The layered-security approach centers on maintaining appropriate security measures and procedures at five different levels within an IT environment:

  1. Perimeter
  2. Network
  3. Host
  4. Application
  5. Data

In the context of SOA based integration, the technology layer requires and recommends that the messaging systems of business partners that are involved in consuming and providing business critical services are fully secured by preventing unauthorized users or systems from deliberately or accidentally accessing the business services.

In the context of SOA based integration, security could be applied at two different layers as follows:

  • Point-to-Point Transport Level Security
  • End-to-End Business Message Level Security

In most intra-enterprise business scenarios, transport level security is sufficient. In some inter-enterprise business scenarios, especially when an intermediary node is involved, business message level security is desired.

Following are the security features that could be implemented or deployed for securing business services, depending on specific business scenarios and requirements:

  • Authentication
  • Authorization
  • Confidentiality
  • Integrity
  • Non-Repudiation

From the ‘standards’ and ‘best practices’ perspective, for inter-enterprise services, SSL is preferred for transport level security and WS-Security is preferred for message level security. SSL offers the security features of authentication, confidentiality and integrity at the transport level, whereas WS-Security offers all of the above-mentioned security features at the message level.

Authentication Patterns

Authentication is the process of identifying an individual or system using the credentials of that individual or system.

Following are the 2 high level patterns of authentication:

  1. Direct Authentication
  2. Brokered Authentication

Both the Direct Authentication pattern and the Brokered Authentication pattern focus on the relationships that exist between a client and service participating in a Web service interaction.

When both the client and service participate in a trust relationship, that allows them to exchange and validate credentials including passwords, direct authentication can be performed. Direct authentication requires the presentation of credentials, which are typically a user name and password. The service uses these credentials to authenticate the request directly.\

In a situation where the client and service do not share a direct trust relationship, broker authentication can be performed. Credentials are used to authenticate with the broker, which issues a security token. The security token is then used to authenticate with services.

Depending on the type of token, security protocol standard etc, following are the 3 kinds of brokered authentication mechanisms that are widely used:

  1. X.509 PKI
  2. Kerberos
  3. Security Token Service (ex, SAML)
Authorization Patterns

Following are the 2 different methods of performing authorization:

  • Role-based
  • Resource-based

Role-based authorization is used to associate clients and groups with the permissions that they need to perform particular functions or access resources. When a user or group is added to a role, the user or group automatically inherits the various security permissions. Role-based authorization can be declarative or imperative.

Resource-based authorization is performed declaratively on a resource, depending on the type of the resource and the mechanism used to perform authorization. Resource based authorization can be based on:

  • Access Control Lists (ACLs) or
  • URLs

Individual resources are secured using ACLs that specify the type of operation a particular security principal or group to which a security principal belongs can perform on the object.

URLs are declared with permissions for the URL to define who is authorized to access the URL in question.

Message Protection Patterns

Web services send and receive plaintext messages over standard Internet protocols such as HTTP. Such plaintext messages can be intercepted by an attacker and potentially viewed or even modified for malicious purposes. By using message protection, you can protect sensitive data against threats such as eavesdropping and data tampering.

Message protection can be divided into three main categories:

  • Data Integrity
  • Data Origin Authentication
  • Data Confidentiality

Data integrity is the verification that a message has not changed in transit.

Data Origin Authentication takes data integrity a step further and supports the ability to identify and validate the origin of a message.

Data Confidentiality is the encrypting of message data so that unauthorized entities cannot view the contents of the message. Data confidentiality can be applied in two steps:

  1. Encrypting the data. In this step, the sender converts plaintext to cipher text, rendering it unintelligible to parties other than the intended recipient.
  2. Decrypting the data. In this step, cipher text is rendered intelligible to the intended recipient by converting it back to plaintext.

Two types of cryptography can be used to provide data confidentiality: symmetric or asymmetric.

With symmetric cryptography, both the sender and recipient share a key that is used to perform both encryption and decryption. Symmetric cryptography is commonly used to perform encryption. It also provides data integrity when symmetric keys are used in conjunction with other algorithms to create Message Authentication Codes (MACs). Some of the more common symmetric cryptography algorithms include Rijndael (AES) and Triple DES (3DES). These algorithms are designed to perform efficiently on common hardware architectures.

Symmetric cryptography is comparatively simple in nature, because the secret key that is used for both encryption and decryption is shared between the sender and the recipient. However, before communication can occur, the sender and the recipient must exchange a shared secret key. In some cases (such as SSL), asymmetric cryptography can be used to ensure that the initial key exchange occurs over a secure channel.

With asymmetric cryptography (also known as public key cryptography), the sender encrypts data with one key, and the recipient uses a different key to decrypt ciphertext. The encryption key and its matching decryption key are often referred to as a public/private key pair. The public key of the recipient is used to encrypt data. It can be openly distributed to those who want to encrypt a message to the recipient. The private key of the recipient is used to decrypt messages, and only the recipient must be able to access it. The most commonly used asymmetric algorithm is the RSA algorithm.

In the context of SOA for integration, it is important to agree on the cryptography algorithms to be used between service providers and consumers.

Data Origin Authentication enables the recipient to verify that messages have not been tampered with in transit (data integrity) and that they originate from the expected sender (authenticity).

In cases where the client denies having performed the action (non-repudiation), you can use digital signatures to provide evidence that a client has performed a particular action that is related to data. Digital signatures can be used for non-repudiation purposes, but they may not be sufficient to provide legal proof of non-repudiation. By itself, a digital signature is just a mechanism to capture a client’s association to data. In cases where data has been digitally signed, the degree to which an individual or organization can be held accountable is established in an agreement between the party that requires digital signatures and the owner of the digital signature.

Two types of signatures can be used to sign a message: symmetric and asymmetric.

A symmetric signature is created by using a shared secret to sign and verify the message. A symmetric signature is commonly known as a Message Authentication Code (MAC). A MAC is created by computing a checksum with the message content and the shared secret. A MAC can be verified only by a party that has both the shared secret and the original message content that was used to create the MAC.

The most common type of MAC is a Hashed Message Authentication Code (HMAC). The HMAC protocol uses a shared secret and a hashing algorithm (such as MD5, SHA-1, or SHA-256) to create the signature, which is added to the message. The message recipient uses the shared secret and the message content to verify the signature by recreating the HMAC and comparing it to the HMAC that was sent in the message.

An asymmetric signature is processed with two different keys; one key is used to create the signature and the other key is used to verify the signature. The two keys are related to one another and are commonly referred to as a public/private key pair. The public key is generally available and can be distributed with the message; the private key is kept secret by the owner and is never sent in a message. A signature that is created and verified with an asymmetric public/private key pair is referred to as a digital signature.

The algorithm that is most commonly used to create a digital signature is the Digital Signature Algorithm (DSA). DSA uses the public/private key pairs created for use with the RSA algorithm to create and verify signatures.

For both signing and encryption purposes, asymmetric keys are often managed through a Public Key Infrastructure (PKI). Information that describes the client is bound to its public key through endorsement from a trusted party to form a certificate. Certificates allow a message recipient to verify the private key in a client’s signature using the public key in the client’s certificate.

Resource Access Patterns

Web services represent a programmable interface that applications often use to access resources, such as the local file system, databases, or other Web services. Distributed applications can consist of multiple interacting Web services, so you have to consider the needs and constraints of the entire application instead of focusing on a single point of interaction. Authentication, authorization, and auditing, along with other environmental and operational requirements (such as scalability requirements at each resource access interface), should combine to influence the security solution.

The following methods can be used to access resources:

  • Impersonation
  • Delegation
  • Protocol Transition
  • Trusted Subsystem

Impersonation is the act of assuming a different identity on a temporary basis so that a different security context or set of credentials can be used to access a resource. When accessing local resources, such as a file in the local file system, you need only the security context. However, when accessing resources that require authentication, such as a Web service or database, credentials are required.

Delegation is not the same as impersonation; however, it requires impersonation to work. This is a process where the service account is allowed to access a remote resource on behalf of another account, which is typically the client accessing a service. Impersonation is required so that the service account can access the credentials of the account being delegated and send those credentials to the remote resource. For delegation to work, the account being delegated must also be configured to allow delegation.

Protocol transition is a process where the service account transitions an identity that was authenticated using a non-native protocol into a native security context. This can also be used to implement a trusted subsystem by using the service account’s identity instead of the client’s identity to access resources.

Trusted Subsystem is a process where a trusted business identity is used to access a resource on behalf of the client. The identity could belong to the service account or it could be the identity of an application account created specifically for access to remote resources. There are many different reasons for using a trusted subsystem, but the most common reason is to take advantage of resource sharing techniques, such as connection pooling associated with database connections.

Service Boundary Protection Patterns

‘Message Protection Patterns’, described how to provide protection against data tampering and unauthorized access to message content. However, in many cases you will need to provide additional protection at the service’s boundary to:

  • Protect Web services against malformed or malicious content.
  • Ensure that when a Web service operation fails you do not accidentally reveal confidential information in the Fault message that is returned.
  • Prevent an attacker from intercepting a message and replaying it to force a Web service operation to execute multiple times.

Service Boundary Protection includes the following:

  • Message Replay Detection
  • Message Validator
  • Exception Shielding

Message Replay Detection: When the Web service processes the messages, data is updated or business processes are initiated. If a message that is intended for one of these Web services is intercepted and replayed, there is a risk that the same operation might be performed multiple times. Solution is to cache an identifier for incoming messages, and use message replay detection to identify and reject messages that match an entry in the replay detection cache. Message replay detection requires that individual messages can be uniquely identified.

Message Validator: A Web service interacts with other applications over a network. Incoming data may be malformed and may have been transmitted for malicious purposes. There is also a risk of injection attacks, where data from incoming messages is tampered with to include additional syntax. Assume that all input data is malicious until proven otherwise, and use message validation to protect against input attacks, such as SQL injection, buffer overflows, and other types of attacks. The message validation logic enforces a well-defined policy that specifies which parts of a request message are required for the Web service to successfully process it. It validates the XML message payloads against an XML schema (XSD) to ensure that they are well-formed and consistent with what the Web service expects to process. The validation logic also measures the messages against certain criteria by examining the message size, the message content, and the character sets that are used. Any message that does not meet the criteria is rejected.

Exception Shielding: Use the Exception Shielding pattern to sanitize unsafe exceptions by replacing them with exceptions that are safe by design. Return only those exceptions to the client that have been sanitized or exceptions that are safe by design. Exceptions that are safe by design do not contain sensitive information in the exception message, and they do not contain a detailed stack trace, either of which might reveal sensitive information about the Web service’s inner workings.

Oracle AS10g and BPEL PM provides most of the common security infrastructure and OWSM provides a policy driven security features for advanced deployments. The above patterns must be considered for a given service before deciding on using the Oracle SOA Suite components.

FAULT RAISING AND HANDLING

In service-oriented messaging infrastructure, faults or exceptions can occur either at transport level or message processing level (such as SOAP) or business service level.

It is important to handle every fault scenario to build robust and predictable services while simultaneously providing visibility into the state of the system.

Broadly, exceptions or faults can be classified horizontally into the following categories:

  • Unrecoverable Exceptions: Unrecoverable exceptions are unexpected fatal error conditions within a system that cannot be handled gracefully by the involved systems or services automatically and require manual handling.
  • Recoverable Exceptions: Recoverable exceptions denote that some condition in the system has not been met for a given context but may be recoverable gracefully through alternative means by the involved systems or services automatically.

Again, broadly, exceptions can further be classified vertically into the following categories:

  • System Exceptions: System exceptions are hardware or software related issues that are related to IT organization of an enterprise. Some of the system exceptions are recoverable and most of them are unrecoverable automatically and requires manual handling.
  • Business Exceptions: Business exceptions are part of the business service interface and implementation contract having business semantics. These exceptions are expected by the involved parties (i.e., service producers and service consumers) and hence are recoverable through an alternative business process flow or human workflow.

Following are the system exception scenarios that can occur in a service-oriented system and the corresponding processing patterns:

  • System Resource Exceptions: System resource exceptions indicate a fatal situation related to system resources. System Crash, Resource Unavailability, Out-of-Memory exception, etc. could fall under this category. These types of exceptions can either be handled manually or automatically. By acquiring an alternative resource, systems can automatically recover from these exceptions. For example, in case of resource unavailability, fail over to an alternative resource or retry mechanisms might allow the system to recover. But typically, resource exceptions are handled manually by restarting or reallocating resources.

    In case of unrecoverable exceptions, such as System Crash or Out-of-Memory exceptions, ‘networking and system management’ tools must be used to detect the situation and raise an alert to the involved parties. System Resource Exceptions must be raised and alert both business users as well as IT support personnel so that all the involved parties are aware of the system as well as business state. But System Resource Exceptions should be handled only by the IT support personnel.

    Synchronous MEP based Services will throw system resource exceptions back to the consumer at the time of consuming the service as a fault message.
  • Internal System Exceptions: Internal system exceptions occur usually due to programming logic errors on the service provider side such as null pointer exceptions. Usually these exceptions occur when the provider is not cooperating to handle that particular interaction from the consumer’s perspective. Ideally these exception conditions should never occur. But practically it is impossible to avoid these exceptions.

    The service consumers cannot handle exceptions on the provider side and hence optionally exceptions may be raised to the consumer for visibility purposes, but the exception must be handled by the provider only. Exceptions must be logged and the provider must raise an appropriate alert for visibility purposes. Since it is an internal problem, the same message can be processed again after fixing the problem either automatically or manually. Internal System Exceptions must be raised and alert both business users as well as IT support personnel so that all the involved parties are aware of the system as well as business state. Internal System Exceptions should either be handled by the IT support personnel or business users or both depending upon the situation.

    Synchronous MEP based Services will throw internal system exceptions back to the consumer as a fault message.
  • External System Exceptions: External system exceptions occur usually due to programming logic errors on the consumer side. Usually these exceptions occur when the consumer violates the interaction agreement from the provider’s perspective. Ideally these exception conditions should never occur. But practically it is impossible to avoid these kinds of exceptions.

    Exceptions due to consumer error should be thrown to the consumer by the provider so that it may be handled by the consumer by taking alternative actions and if not possible, exceptions must be logged and an appropriate alert must be raised for visibility purposes by the consumer. External System Exceptions must be raised and alert both business users as well as IT support personnel so that all the involved parties are aware of the system as well as business state. External System Exceptions should either be handled by the IT support personnel or business users or both depending upon the business requirements.

    Synchronous MEP based Services will throw external system exceptions back to the consumer as a fault message.

The details of raising and handling faults at every service-oriented messaging layer must be defined and an example is provided below:

Transport Layer

The fault bindings for the transports used from the consumer’s perspective is defined below:

  • HTTP
  • System Resource Exception
    • 404 Not Found
    • 503 Service Unavailable
  • Internal and External System Exception
    • 500 Internal Server Error

SOAP Layer

Exceptions are propagated to the service consumer using the SOAP1.1 Fault message. A SOAP1.1 fault message is an envelope that has a <soap:Fault> element as single child element of the <soap:Body> element and the <soap:Fault> element must not have element children other than <faultcode>, <faultstring>, <faultactor> and <detail>.

In case of HTTP transport binding, ‘500 Internal Server Error’ status code must be used to carry the soap fault message.

Business Service Layer

Business faults must be propagated to the business service consumer as a literal document in a SOAP1.1 envelope.

Synchronous MEP based services will raise all the faults to the service consumer as SOAP Faults. The <detail> element of the SOAP Fault will carry the detailed Fault message.

From the BPEL perspective, throw and catch activities and fault handlers must be used appropriately. Also BPEL based human workflow process can be used to handle exceptions manually.

Fault handling in long running transaction scenarios might involve compensation handlers to bring the system back to a consistent state.

WS-I COMPLIAN WSDL TEMPLATES

The Web Services Interoperability Organization is an open industry effort chartered to promote Web services interoperability across platforms, applications, and programming languages. The organization brings together a diverse community of Web services leaders to respond to customer needs by providing guidance, recommended practices, and supporting resources for developing interoperable Web services.

Section 5 of the WS-I Basic Profile 1.0 specification relates to conformant use of WSDL to describe a Web service. Of all the sections in the profile specification, it is probably of most interest and importance to Web services developers. It begins by addressing issues related to WSDL document structure and composition.

The first issue addressed is that of correct usage of the wsdl:import feature. The wsdl:import feature is intended to be used in the same manner as the xsd:import feature, to import definitions from a foreign namespace. e.g. a namespace that is not the same as the value of the targetNamespace attribute of the importing WSDL document. Additionally, the WS-I Basic Profile constrains use of the <wsdl:import> feature to the import of WSDL definition components from another WSDL document. This means that you cannot use the <wsdl:import> feature to import schema definitions from an XSD file.

The profile also constrains the order in which the top-level WSDL elements are to be used. Specifically, it requires that the wsdl:import element, if present, precede all other top-level WSDL elements. It is then followed by the <wsdl:types> element, and then <wsdl:message>, <wsdl:portType>, <wsdl:binding> and <wsdl:service> elements which may appear as many times as needed, and in any order.

This constraint, like many others in the specification, is based on decisions made by the W3C Working Group defining the next version of the Web services specifications for SOAP and WSDL. The reason that the Basic Profile WG has adopted certain decisions made by the W3C WGs is that they wanted to resolve identified interoperability issues in a manner consistent with the direction of the W3C WG drafting the 1.2 versions of the SOAP and WSDL specifications. This approach is intended to facilitate a smoother transition to the next version of the specifications once they reach W3C Recommendation status and are incorporated into subsequent WS-I Profiles.

SOAP Encoding Style

A WSDL binding describes how the service is bound to a messaging protocol, particularly the SOAP messaging protocol. WSDL 1.1 distinguishes between two soap message binding styles: document and RPC. Here's how each style affects the contents of <SOAP:Body>.

  • Document: <SOAP:Body> contains one or more child elements called parts. There are no SOAP formatting rules for what the <SOAP:Body> contains; it contains whatever the sender and the receiver agree upon.
  • RPC: RPC implies that <SOAP:Body> contains an element with the name of the method or remote procedure being invoked. This element in turn contains an element for each parameter of that procedure.

For applications that use serialization/deserialization to abstract away the data wire format, there's one more choice to be made: the serialization format. There are two popular serialization formats today:

  • SOAP Encoding: SOAP encoding is a set of serialization rules defined in section 5 of SOAP 1.1 and is sometimes referred to as "section 5 encoding." The rules specify how objects, structures, arrays, and object graphs should be serialized. Generally speaking, an application using SOAP encoding is focused on remote procedure calls and will likely use RPC message style.
  • Literal: Data is serialized according to a schema. In practice, this schema is usually expressed using W3C XML Schema. Although there are no prescribed rules for serializing objects, structures, and graphs, etc., the service's schema (usually XSD) describes the application-level Infoset of each of the service's messages.

Thus it gives us 4 different choices of soap binding styles. In addition to these 4 styles, a 5th binding style, which is commonly called the “document/literal wrapped”, is also available to choose. WS-I Basic Profile 1.0 disallows encoded content due to the complexity and incomplete support for the spec by various toolkits; only Document/Literal and RPC/Literal are allowed.

Document/Literal is highly interoperable over other styles (hence recommended by WS-I):

  • Messaging oriented as opposed to RPC
  • Enables loosely-coupled service implementation instead of tightly coupled
  • Content within the <SOAP:Body> can be easily validated against a XML Schema (XSD)
  • .NET Platform uses Document/Literal style as its default

Whereas the RPC/Encoded style has the following weaknesses:

  • Encourages tightly coupled service implementation
  • The type encoding info (such as xsi:type) is an overhead, degrading throughput performance
  • Message validation using XSD cannot be easily validated

In essence, there are two compelling reasons to use document-style messaging. One is to gain the independence it provides. Its strength lies in decoupling interfaces between services, such as in application-to-application and system-to-system interfaces, to the point that they can change completely independently of one another. The other is that document-style messaging puts the full power of XML for structuring and encoding information at our disposal.
