OUM

Relationship to Unified Process (UP)

OUM is based on non-proprietary standards like Unified Modeling Language (UML) and de facto standards like the Unified Process (UP). OUM relies on the Unified Process concepts and principles, like incremental delivery, iterative software development, UML-based, architecture-centric, and requirements-driven.

The following table provides an overview of how OUM relates to UP.

OUM Component UP Component Comments
Analysis (process)
 Together with the Technical Architecture process, the OUM Analysis process makes up what in the Analysis core workflow is in UP. The OUM Analysis process does not directly address the overall supplemental requirements.
Business Requirements (process)
 Together with the Requirements Analysis process, the OUM Business Requirements process makes up what the Requirements core workflow is in UP. This process focuses on capturing high-level requirements, both functional as well as supplementary. These requirements are captured using a MoSCoW List rather than a use case. The MoSCoW List with functional requirements can be compared to an Actor-Goal List as defined in Writing Effective Use Cases, Alistair Cockburn. For futher reading, see Writing Effective Use Cases.
Testing (process)
 Together with the Performance Management process, the OUM Business Testing process makes up what the Requirements core workflow is in UP. Performance testing requires a specific environment, skills and approach, and therefore is dealt with in OUM by a separate process.
Data Acquisition and Conversion (process)
 Data conversion currently is not properly dealt with by UP. UP mentions data conversion only briefly when describing the Transition phase. As data conversion often requires significant planning, analysis and effort to accomplish, Therefore OUM has a separate process for it.
Work Product
 A work product in OUM is the single result of a task, and may consist of one or more artifacts. An artifact can be part of some other artifact. A work product cannot be part of some other work product.
Design (process)
 The OUM Design process is similar to the UP Design core workflow.  
Implementation (process)
 The OUM Implementation process is similar to the UP Implementation core workflow.  
Performance Management (process)
 Together with the Testing process, the OUM Performance Management process makes up what the Test core workflow is in UP. See also the note for the Testing process.
Phase
 The phases in OUM are equal to those in UP, except for the fact that OUM also recognizes a Production phase.  
Operations and Support (process)
 Like UP has no Production phase, it has no process that can be compared to the OUM Operations and Support process either.  
Production (phase)
 UP stops after the Transition phase and does not describe a production phase. For the Transition phase UP only mentions that the next release or generation of the system should be planned. OUM recognizes that after bringing the system into production, support must be provided, the system must be monitored, and enhancements must be defined and planned.
Process
 A process in OUM is what a core workflow is in UP.  
Requirements Analysis (process)
 Together with the Business Requirements process, the OUM Requirements Analysis process makes up what is in UP, the Requirements core workflow. The analysis of requirements addressing specific overall supplementary requirements issues is dealt with in the Technical Architecture process. Within this process the requirements that have been captured during the Business Requirements Definition process as a MoSCoW List, are analyzed and described as use cases. Also the use cases details are created within the Requirements Analysis process.
Task
 A task is what an activity is in UP in the context of a unit of work resulting in a well-defined set of results (artifacts).  
Technical Architecture (process)

Together with the Analysis process, the OUM Application and Technical Architecture makes up what the Analysis core workflow is in UP.

 Technical Architecture covers generic supplementary requirements, like fallback and recovery, security and control, hardware and software components, and capacity and performance. Software components in this context concern for example operating systems, network software, Oracle tools, database servers, or frameworks.
Transition (process)
 UP does not recognize an explicit Transition core workflow. As the Transition phase needs to be planned and documented in an early stage, OUM has a separate process for it. UP does recognize the need for planning the transition but does not identify an artifact that can be compared with the Cutover Strategy of OUM.
Role
 A role in OUM is what a worker is in UP. In UP only the worker that is responsible for the activity is indicated, whereas in OUM all roles that participate in performing the task are listed.
